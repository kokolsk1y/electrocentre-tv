## 1. Краткие выводы

- **Ядро работает.** Слайдер с 9 слайдами (4 HTML + 5 баннеров), progress-bar, часы, бегущая строка — всё функционирует. Это не прототип, это уже рабочий продукт.
- **Критический риск: внешние зависимости при запуске.** Google Fonts подключены по CDN (`fonts.googleapis.com`). Если Wi-Fi в магазине медленный или нестабильный — первый рендер будет с системным шрифтом или зависнет. Это единственная сетевая зависимость, но она критична для фирменного стиля.
- **Переход между слайдами — белая вспышка.** Механизм `.fl.on { opacity:1 }` с задержкой 120ms создаёт заметный белый флэш. На TV с высокой яркостью это агрессивно. Замена на cross-fade — первоочередная задача.
- **Имена файлов баннеров — кириллица и пробелы** (`2слайд_upscayl_2x_remacri-4x.png`). На некоторых серверах и TV-браузерах это создаёт проблемы с URL-кодированием. Переименование обязательно до добавления новых баннеров.
- **Все слайды одинаковой длительности (10 сек).** HTML-слайды с несколькими элементами анимируют их с задержкой 900ms между каждым — на слайдах с 5+ элементами последний появляется на 4-й секунде. При 10 секундах это нормально. При уменьшении длительности анимация не успеет отработать.
- **Нет защиты от burn-in.** TV в магазине работает весь рабочий день. Header и footer статичны и занимают фиксированные позиции — долгосрочный риск выгорания матрицы на OLED/QLED.
- **Управление контентом — только через код.** Нет ни одного механизма для обновления контента без правки HTML. Это осознанный компромисс v1, но он блокирует самостоятельность команды магазина.

## 2. Критические исправления

Это то, что нужно сделать до любых новых функций.

### 2.1 Встроить шрифт Montserrat как base64 или self-hosted

**Проблема:** `<link href="https://fonts.googleapis.com/...">` — блокирующий внешний запрос при старте.

**Решение:** Скачать woff2-файлы нужных начертаний (300, 400, 600, 700, 800, 900) и встроить через `@font-face` с `src: url('data:font/woff2;base64,...')` прямо в `<style>`. Файл вырастет на ~200-400 KB, но станет полностью автономным.

**Приоритет: Высокий.** Это условие из PROJECT.md: "Работает в браузере Smart TV без плагинов" — нестабильный Wi-Fi нарушает это требование.

### 2.2 Переименовать файлы баннеров в ASCII

**Проблема:** `slides/2слайд_upscayl_2x_remacri-4x.png` — кириллица в именах файлов, которые идут в `src=""` атрибутах.

**Решение:** Переименовать по схеме `banner-01.webp`, `banner-02.webp` и т.д. Обновить ссылки в HTML. Сделать это один раз сейчас, пока баннеров 5.

**Приоритет: Высокий.** Любой новый TV-браузер или будущий хостинг может сломаться на кириллице в URL.

### 2.3 Заменить белый флэш на cross-fade

**Проблема:** `.fl { background:#fff }` + `fl.classList.add('on')` создаёт белую вспышку при каждом переходе.

**Решение:** Вместо flash-overlay использовать `opacity: 0 → 1` на самом слайде с `position:absolute`. Оба слайда сосуществуют 300-400ms: уходящий затухает, входящий проявляется. Реализуется через добавление классов `.leaving` и `.entering` с CSS transition.

**Приоритет: Средний** (визуальный дефект, не функциональный баг).

## 3. Рекомендации по разработке

Упорядочены по приоритету и зависимостям.

### Фаза 1 — Стабилизация (сделать до показа руководству)

1. **Self-hosted шрифт** (см. 2.1)
2. **ASCII-имена баннеров** (см. 2.2)
3. **Cross-fade переход** (см. 2.3)
4. **Скринсейвер при простое** — после 17:30 (или N минут бездействия) переходить в режим с движущимся логотипом. Это критично для TV с OLED-панелью. Реализуется через проверку `new Date().getHours()` в том же JS.

### Фаза 2 — Контентное улучшение

5. **Разные длительности слайдов** — HTML-слайды 10 сек, баннеры 12-15 сек. Механизм `data-duration` уже есть в коде — нужно только расставить значения.
6. **Дата в шапке** — рядом с часами, те же 3 строки JS. Низкие риски, высокая воспринимаемая ценность.
7. **QR-код на баннерных слайдах** — небольшой QR в углу каждого баннера, ведёт на stv39.ru или страницу акции. Генерировать статично (не в JS), вставлять как PNG в баннер при подготовке — это проще и надёжнее, чем qrcode.js на TV.

### Фаза 3 — Управление контентом (требует решения по архитектуре)

8. **Авто-обновление баннеров через GitHub** — TV читает `manifest.json` из GitHub Pages раз в час через `fetch()`, подгружает новые URL. Менеджер обновляет JSON вручную или через GitHub UI. Это минимальная CMS без сервера.
   - Зависимость: стабильный Wi-Fi в магазине (см. раздел 5).
   - Риск: `fetch()` заблокирован на некоторых Smart TV браузерах — нужно проверить на реальном TV.

9. **Google Sheets как конфиг слайдов** — альтернатива manifest.json, более удобна для нетехнического менеджера. TV читает опубликованный CSV через Sheets API. Минус: зависимость от Google, сложнее отлаживать.
   - Рекомендация: выбирать между п.8 и п.9 после теста TV на поддержку fetch(). Если fetch() работает — manifest.json проще. Если нет — Google Sheets тоже не поможет.

### Фаза 4 — Аудио (отдельная тема)

10. **Фоновая музыка** — в браузерах Smart TV `autoplay` для аудио заблокирован без пользовательского жеста. На Smart TV без пульта это неразрешимая проблема стандартными средствами. Нужно исследовать, разрешает ли конкретный TV autoplay для локальных файлов или kiosk-режима.
11. **TTS через Яндекс SpeechKit** — требует API-ключ и сетевой запрос. Нельзя хранить ключ в публичном HTML на GitHub Pages. Нужен прокси (n8n или Cloudflare Worker). Реализовывать только после решения проблемы autoplay.

## 4. Что НЕ делать

- **Не добавлять новые функции поверх незакрытых критических исправлений.** Белый флэш и кириллица в именах файлов должны быть исправлены до следующего показа.
- **Не встраивать API-ключи в HTML.** Файл публичен на GitHub Pages. Любой ключ (SpeechKit, погода, аналитика) станет публичным немедленно.
- **Не полагаться на fetch() без проверки.** Smart TV браузеры (особенно на устаревших моделях Samsung/LG) могут блокировать fetch() к сторонним доменам или вообще не поддерживать Promise-based API.
- **Не использовать кириллицу в именах файлов, путях и URL.** Ни в HTML-атрибутах, ни в именах на диске.
- **Не добавлять тяжёлые JS-библиотеки** (React, Vue, jQuery). Текущий ванильный JS работает быстро и без зависимостей — это преимущество, не недостаток.
- **Не создавать v14, v15 и т.д.** Согласно правилам проекта — только редактировать v13.
- **Не реализовывать аналитику просмотров** (задача #36) пока неизвестна модель TV и нет прокси для безопасной отправки данных.
- **Не использовать CSS-анимации с фиксированной частотой обновления** без проверки на TV — некоторые Smart TV ограничивают requestAnimationFrame до 30fps.
